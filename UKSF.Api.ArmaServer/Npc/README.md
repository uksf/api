# NPC Speech Broker (sub-plan #5)

This directory contains the C# implementation of the NPC speech broker. Inbound events arrive on the existing `/gameservers/events` channel (`npc_register`, `npc_turn`); outbound audio commands are POSTed as SQF-array literals to `http://localhost:{apiPort}/command` on the game server; outbound brain calls go to the local clacks daemon ({CLACKS_URL}/chat for routing, /speak for TTS); prerendered + dynamic clips are stored as WAV files under {NPC_AUDIO_PATH} with paths in Mongo. Session and cached-audio state persist in Mongo collections `npcSessions` and `npcAudioClips` so a mid-mission API restart loses nothing. The entire feature is gated by the `FEATURE_NPC_BROKER` variable.

---

## Required `variables` collection entries (operators must add these)

| Key | Type | Example value | Purpose |
|-----|------|---------------|---------|
| `FEATURE_NPC_BROKER` | bool | `true` | Gates the entire feature. Read via `variablesService.GetFeatureState("NPC_BROKER")`. |
| `NPC_AUDIO_PATH` | string | `D:/NpcAudio` | Root folder for WAV clips (date-foldered). Mongo npcAudioClips stores paths relative to this root. Files survive mission end as the session voice-line archive. |

## #5 ↔ #6 wire contract

### Game → API (inbound events, emitted by #6)

**`npc_register`** — once per talkable NPC at mission start.

`Data` shape:
```json
{
  "npcId": "string",
  "sessionId": "string",
  "persona": {
    "name": "string",
    "role": "string",
    "language": "string",
    "mood": "string",
    "attitudeToPlayers": "string"
  },
  "knowledge": "string",
  "mode": "scripted | dynamic",
  "scripted": {
    "lines": [{ "id": "string", "topic": "string", "line": "string" }],
    "deflection": "string"
  },
  "voiceId": "string"
}
```

`scripted` is present only when `mode = "scripted"`. `apiPort` is NOT in `Data` — it rides the event envelope as `gameServerEvent.ApiPort`.

---

**`npc_turn`** — one batched turn after the room debounce fires.

`Data` shape:
```json
{
  "npcId": "string",
  "sessionId": "string",
  "turnId": "string",
  "newTurns": [{ "speakerId": "string", "text": "string", "t": 1700000000000 }]
}
```

`turnId` is generated by #6 and is used to correlate the pushed audio. `apiPort` rides the event envelope.

---

### API → game (outbound commands, consumed by #6)

Commands are POSTed as `text/plain` SQF-array literals to `http://localhost:{apiPort}/command`. The extension listener rejects bodies larger than 65536 bytes (HTTP 413), so base64 audio is split into chunks and each command stays under 64 KB. #6 reassembles chunks by `(npcId, turnId)` or `(npcId, fillerId)`, base64-decodes the concatenated payload, and plays audio anchored to the NPC netId.

**`npc_filler`** — pushed once after register so clients can cache filler clips and play one instantly on utterance-finalise. Random filler selection is #6's responsibility.

```
["npc_filler", npcId, voiceId, fillerId, index, total, "<base64chunk>", durationMs]
```

Fixed filler set (pre-rendered in each NPC's `voiceId`):

| id | text |
|----|------|
| `f0` | `hmm` |
| `f1` | `let me think` |
| `f2` | `give me a sec` |
| `f3` | `hold on` |

---

**`npc_audio`** — per-turn audio. Both scripted-from-cache and dynamic-fresh responses use this shape.

```
["npc_audio", npcId, turnId, index, total, "<base64chunk>", durationMs]
```

`index` and `total` are 0-based chunk counters. Concatenate all chunks in order, base64-decode the result, then play the Opus audio anchored to the NPC netId.
